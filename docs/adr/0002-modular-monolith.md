# ADR-0002: 모듈러 모놀리스 아키텍처

- **Status**: Accepted
- **Date**: 2026-04-17
- **Deciders**: wanny

## Context

7일 내 OTA 플랫폼 백엔드 설계 및 구현을 수행한다. 이때 도메인 주도 설계(DDD) 기반의 바운디드 컨텍스트를 시각화하고 구현해야 한다.

평가 기준 중 C축(도메인 설계)에서 아키텍처 경계를 코드와 테스트로 명확하게 증명해야 한다. 따라서 아키텍처 구조가 코드 레벨에서 명시적으로 드러나야 하고, 경계 간 의존성 방향을 강제할 수 있어야 한다.

제약 조건:
- 7일의 제한된 일정
- 단일 팀(1명) 구성으로 운영 단순성 필요
- 초기 배포는 단일 실행 파일(JAR)로 제공

## Decision

**단일 Gradle 모듈 내에서 패키지 단위로 바운디드 컨텍스트를 분리하는 모듈러 모놀리스 아키텍처를 채택한다.**

구체적으로:
- 프로젝트는 `src/main/kotlin/global/ota/` 하에 단일 모듈로 구성
- 각 바운디드 컨텍스트는 `global.ota.<context>/` 패키지 계층으로 분리
- ArchUnit을 도입하여 패키지 간 의존 방향 규칙을 테스트로 강제
- 런타임은 단일 Spring Boot 애플리케이션으로 동작

## Alternatives Considered

### Option A: 규율 없는 단일 모놀리스

**장점:**
- 가장 빠른 착수 (패키지 구조 설계 최소화)
- 모든 기능이 하나의 패키지 트리에 자유롭게 위치

**단점:**
- 코드에서 아키텍처 경계가 명시되지 않음
- 패키지 구조가 임의적이어서 평가자가 도메인 설계를 읽을 수 없음
- C축 증명 근거 불충분
- 시간이 지날수록 경계가 무너져서 유지보수 비용 증가

**선택하지 않은 이유:** 평가 기준 C축(도메인 설계)을 증명할 수 없음

### Option B: 모듈러 모놀리스 + ArchUnit (선택)

**장점:**
- 패키지 경계를 코드로 시각화하여 C축 증명 명확
- 단일 JAR 배포로 초기 구현 속도 유지
- ArchUnit 테스트로 경계 위반을 자동 감지
- 향후 트래픽/팀 규모 증가 시 물리 모듈(multi-gradle-module)로 승격 경로 명확
- 7일 기간 내에 현실적으로 구현 가능

**단점:**
- ArchUnit 규칙을 꾸준히 유지해야 경계가 무너지지 않음 (개발 규율 필요)
- 런타임에는 물리적 분리가 없어서 한 컨텍스트의 장애가 전체에 영향

**선택 이유:** 아키텍처 명확성과 7일 일정의 최적 균형

### Option C: 멀티 Gradle 모듈 (각 컨텍스트별 독립 모듈)

**장점:**
- 물리적 경계를 강제하여 컴파일 레벨에서 의존성 위반 방지
- 명확한 공개 API(모듈 export)

**단점:**
- 초기 Gradle 멀티 모듈 구성에 1~2일 소요
- 모듈 간 빌드 설정(버전 관리, 플러그인)이 복잡
- 로컬 개발 시 rebuild 시간 증가
- 7일 기간 대비 셋업 비용 과다

**선택하지 않은 이유:** 초기 셋업 복잡도 대비 7일 일정이 제약

## Consequences

### Positive
- 코드 레벨에서 바운디드 컨텍스트 경계 명시적 표현
- 단일 JAR 파일 배포로 초기 운영 단순화
- ArchUnit 테스트로 아키텍처 규칙 자동 검증
- 향후 물리 모듈로의 점진적 승격 경로 확보
- 개발자 경험: 단일 프로젝트 셋업, 빠른 hot reload

### Negative / Trade-offs
- 런타임에 물리적 격리가 없어서 한 컨텍스트의 버그나 성능 문제가 전체 시스템에 영향
  - 예: Reservation 모듈의 메모리 누수 → 전체 애플리케이션 영향
- ArchUnit 규칙이 개발자 규율에 의존
  - 만약 규칙 유지가 느슨해지면 경계가 시간에 따라 무너짐
- 동시성 제어 및 리소스 격리가 애플리케이션 코드 레벨에서 구현되어야 함

### Follow-up / Revisit Triggers
1. **트래픽 폭증 (동시 사용자 >1000):** 특정 컨텍스트(예: Pricing)를 독립 마이크로서비스로 분리 검토
2. **팀 규모 확대 (개발자 >3명):** 멀티 Gradle 모듈로 승격하여 물리 경계 강제
3. **장애 격리 필요성 증대:** 컨텍스트별 독립 프로세스/컨테이너로 전환
4. **ArchUnit 유지 비용 증가:** 규칙이 복잡해지면 코드 리뷰 기준으로 전환 검토
